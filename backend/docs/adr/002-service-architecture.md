# ADR-002: Архитектура сервисов

## Status

Accepted

## Context

Необходимо определить единую архитектуру для всех сервисов в монорепо.

## Decision

Использовать слоистую архитектуру (Layered Architecture) с разделением на:

- **api/** - HTTP-слой (роуты, контроллеры, DTO)
- **domain/** - Бизнес-логика (сущности, value objects, интерфейсы репозиториев)
- **application/** - Use-cases (сценарии использования)
- **infrastructure/** - Реализации (БД, внешние сервисы, адаптеры)
- **config/** - Конфигурация
- **lib/** - Внутренние утилиты

## Consequences

### Положительные

- Четкое разделение ответственности
- Легко тестировать (можно мокировать infrastructure)
- Независимость бизнес-логики от инфраструктуры
- Проще добавлять новые use-cases

### Отрицательные

- Больше файлов и директорий
- Необходимость следования структуре
- Потенциальный overhead для простых сервисов
